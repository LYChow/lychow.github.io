Pro Multithreading and Memory Management for iOS and OS X Part of Memory Management (第7周Research课题)

发表于 2016-10-07   |     |   阅读次数

ARC规则

1.不能使用retain/release/retainCount/autorelease

编译器会自动插入上面这些代码.

2.不能使用NSAllocateObject/NSDeallocateObject

alloc实际上就是调用NSAllocateObject函数来生成并持有对象.

3.必须遵守内存管理的方法命名规则

1.以alloc,new,copy,mutableCopy名称开头在返回对象时,必须返回给调用方所应当持有的对象.这在ARC也是适用的.

2.不过ARC追加了一条命名规则.以init开始的方法规则必须是实例方法必须要返回对象.返回的对象应该为id类型或该方法声明类的对象类型,或者是该类的超类和子类类型.

4.不要显示调用dealloc

ARC会自动进行处理,因此不必书写[super dealloc].非写不可编译器会报错

5.不能使用NSZone

NSZone在运行时(编译器OBJC2被设定的环境)中已被忽略.

6.对象型变量不能作为C语言结构体(struct/union)的成员

C语言的规约上没有方法来管理结构体的生存周期,因为ARC把内存管理的工作分配了编译器,所以编译器必须能够知道所管理对象的生命周期.例如局部变量可以根据变量的作用域管理其生命周期,但对于C语言的结构体来说,这个标准就是不可实现的.

要想把对象类型变量加入到结构体中可使用下面的方法:
在变量前添加unsafe_unretained修饰符,附有unsafe_unretained修饰符的变量不属于编译器的内存管理对象

7.显示转换’id’ 和 ‘void *’

id型或对象型变量赋值给void *需要进行特定的转换.如果只想单纯的赋值,则可以使用”__bridge转换”

bridge转换中还有另外两种转换,分别是”bridge_retained转换”和”__bridge_transfer转换”

在MRC中__bridge_retained转换变成了retain

__bridge_transfer被转换的变量所持有的对象,在该变量被赋值给转换目标变量后随之释放.

8.使用@autoreleasepool 块替代NSAutoreleasePool

在ARC中不能使用NSAutoreleasePool类,非用不可编译器会报错

自己生成的对象自己持有

使用一下名称开头的方法名意味着生成的对象只有自己持有:alloc、new、copy、mutableCopy

allocMyObject、newThatObject、copyThis、mutableCopyYourObject也意味着生成并持有对象.

allocate、newer、copying、mutableCopyed(这些除外)

copy方法遵循NSCopying协议并实现copyWithZone:方法生成并持有副本，mutableCopy方法遵循NSMutableCopying协议并实现mutableCopyWithZone:方法生成并持有副本。

引用计数

ARC 背后的原理是依赖编译器的静态分析能力，通过在编译时找出合理的插入引用计数管理代码. 加上 -fno-objc-arc 这个编译参数可以启用手工管理引用计数的模式。

引用计数—>内存块地址—>可追溯到内存块

引用计数表各记录中存有内存块地址，可以从各个记录中追溯到各对象的内存块，即使出现故障导致对象占用的内存块损坏，但只有引用计数没有被损坏，就可以确认各个内存块的位置.在利用工具检测内存泄漏时，引用计数的各记录也有助于检测各对象的持有者是否存在

引用计数是为了解决多个对象强引用M对象的情况？

多个对象强引用M对象利用引用计数方案很好的解决了这个问题，在参数 M 的传递过程中，哪些对象需要长时间使用这个对象，就把它的引用计数加 1，使用完了之后再把引用计数减 1。所有对象都遵守这个规则的话，对象的生命期管理就可以完全交给引用计数了。我们也可以很方便地享受到共享对象带来的好处。

对象回收之后retainCount为什么没有变为0？

那为什么在这个对象被回收之后，这个不确定的值是 1 而不是 0 呢？这是因为当最后一次执行 release 时，系统知道马上就要回收内存了，就没有必要再将 retainCount 减 1 了，因为不管减不减 1，该对象都肯定会被回收，而对象被回收后，它的所有的内存区域，包括 retainCount 值也变得没有意义。不将这个值从 1 变成 0，可以减少一次内存的写操作，加速对象的回收。

Core Foundation 与 Foundation转换

在 ARC 下，我们有时需要将一个 Core Foundation 对象转换成一个 Objective-C 对象，这个时候我们需要告诉编译器，转换过程中的引用计数需要做如何的调整。这就引入了bridge相关的关键字，以下是这些关键字的说明：

  • __bridge: 只做类型转换，不修改相关对象的引用计数，原来的 Core Foundation 对象在不用时，需要调用 CFRelease 方法
  • __bridge_retained：类型转换后，将相关对象的引用计数加 1，原来的 Core Foundation 对象在不用时，需要调用 CFRelease 方法
  • __bridge_transfer：类型转换后，将该对象的引用计数交给 ARC 管理，Core Foundation 对象在不用时，不再需要调用 CFRelease 方法

Calloc和Malloc的使用

虽然保证了附有strong修饰符的id型变量被初始化为nil,但并不能保证附有strong修饰符的id指针型变量被初始化为nil.

id __strong *array =nil;

声明动态数组用指针
NSObject __strong array = nil;

使用calloc函数确保分配的附有__strong修饰符变量的容量占有内存块.

如果使用malloc函数分配内存后可用memset等函数将内存填充为0.

如果只是简单的free(array)函数废弃了数组内存块的情况下,数组各元素所赋值的对象不能再次释放,从而引起内存泄漏.这是因为在静态数组中,编译器能够根据变量的作用域自动插入释放赋值对象的代码,而在动态数组中编译器不能确定数组的生命周期,所以无从处理.

所有权修饰符

__strong 修饰符是id类型和对象类型默认的所有权修饰符.

id类型用于隐藏对象类型相当于C语言的void *.

ARC有效不能使用release方法,但我们的确知道编译器自动插入了release

__weak修饰符

看样子strong修饰符编译器就能完美进行内存管理，但遗憾的是仅仅通过strong修饰符还不能解决循环引用问题。

循环引用容易发生内存泄漏。所谓内存泄漏就是应当废弃的对象在超出其生存周期后继续存在。在持有对象的弱引用时，若该对象被废弃，则此弱引用将自动失效且处于nil.

附有weak修饰符的变量所引用的对象废弃时,将nil赋值给该变量怎样实现的?

(获取键值对应记录)从weak表中获取废弃对象的地址为键值的记录.—->(将记录对应的所有变量赋为nil)将包含在记录中的所有附有__weak修饰符变量的地址,赋值为nil.—->(删除记录)从weak表中删除该记录.—->(引用计数表中删除记录)从引用计数表中删除废弃对象的地址为键值的记录.

__unsafe_unretained修饰符

附有__unsafe_unretained修饰符的变量不属于编译器内存管理的对象，如果没有确保其确实存在，那么应用程序就会崩溃

__autoreleasing修饰符

指定@autoreleasepool块来替代 “autoreleasepool类对象的生成、持有以及废弃”

id的指针或对象的指针在没有显式指定时会被附加上autoreleasing修饰符。虽然可以显式指定autoreleasing修饰符，但在显式指定__autoreleasing修饰符必须注意对象的变量要为自动变量(包括局部变量、函数以及方法参数)

要通过将对象赋值给附加了autoreleasing修饰符的变量来替代autorelease方法，ARC有效时要用@autoreleasepool替代NSAutoreleasePool类，对象赋值给附有autoreleasing修饰符的变量等价于在ARC无效时调用对象的autorelease方法，即对象被注册到autoreleasepool中.

编译器会检查方法名是否以alloc/new/copy/mutableCopy开始，如果不是则自动将返回值的对象注册到autoreleasepool中.

为什么在访问附有weak修饰符的变量时必须访问注册到autoreleasepool的对象呢？

这是因为weak修饰符支持有对象的弱引用，访问该对象的过程中该对象可能被废弃，如果把访问的对象 注册到autoreleasepool中,那么在@autoreleasepool块结束前都能确保对象存在.

__autoreleasing修饰符的变量等同于ARC无效时调用对象的autorelease方法
