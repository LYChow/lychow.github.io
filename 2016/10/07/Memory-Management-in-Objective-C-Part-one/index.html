Memory Management in Objective-C Part one(第7周Research课题)

发表于 2016-10-07   |     |   阅读次数

自己生成的对象自己持有

使用一下名称开头的方法名意味着生成的对象只有自己持有:alloc、new、copy、mutableCopy，allocMyObject、newThatObject、copyThis、mutableCopyYourObject也意味着生成并持有对象，以下除外allocate、newer、copying、mutableCopyed

copy方法遵循NSCopying协议并实现copyWithZone:方法生成并持有副本，mutableCopy方法遵循NSMutableCopying协议并实现mutableCopyWithZone:方法生成并持有副本。

引用计数

ARC 背后的原理是依赖编译器的静态分析能力，通过在编译时找出合理的插入引用计数管理代码. 加上 -fno-objc-arc 这个编译参数可以启用手工管理引用计数的模式。

引用计数—>内存块地址—>可追溯到内存块 //引用计数表各记录中存有内存块地址，可以从各个记录中追溯到各对象的内存块，即使出现故障导致对象占用的内存块损坏，但只有引用计数没有被损坏，就可以确认各个内存块的位置.在利用工具检测内存泄漏时，引用计数的各记录也有助于检测各对象的持有者是否存在

引用计数解决多个对象强引用M对象的情况？

所以引用计数很好的解决了这个问题，在参数 M 的传递过程中，哪些对象需要长时间使用这个对象，就把它的引用计数加 1，使用完了之后再把引用计数减 1。所有对象都遵守这个规则的话，对象的生命期管理就可以完全交给引用计数了。我们也可以很方便地享受到共享对象带来的好处。

对象回收之后retainCount为什么没有变为0？

那为什么在这个对象被回收之后，这个不确定的值是 1 而不是 0 呢？这是因为当最后一次执行 release 时，系统知道马上就要回收内存了，就没有必要再将 retainCount 减 1 了，因为不管减不减 1，该对象都肯定会被回收，而对象被回收后，它的所有的内存区域，包括 retainCount 值也变得没有意义。不将这个值从 1 变成 0，可以减少一次内存的写操作，加速对象的回收。

Core Foundation 与 Foundation转换

在 ARC 下，我们有时需要将一个 Core Foundation 对象转换成一个 Objective-C 对象，这个时候我们需要告诉编译器，转换过程中的引用计数需要做如何的调整。这就引入了bridge相关的关键字，以下是这些关键字的说明：

  • __bridge: 只做类型转换，不修改相关对象的引用计数，原来的 Core Foundation 对象在不用时，需要调用 CFRelease 方法。
  • __bridge_retained：类型转换后，将相关对象的引用计数加 1，原来的 Core Foundation 对象在不用时，需要调用 CFRelease 方法。
  • __bridge_transfer：类型转换后，将该对象的引用计数交给 ARC 管理，Core Foundation 对象在不用时，不再需要调用 CFRelease 方法。

release和autorelease

release立即释放,autorelease不立即释放,注册到autoreleasepool中，pool结束时自动调用release.

如果是多层嵌套的NSAutoreleasePool,理所当然使用最内侧的对象

alloc的底层实现

所有权修饰符

__strong 修饰符是id类型和对象类型默认的所有权修饰符.

id类型用于隐藏对象类型相当于C语言的void *.

__weak修饰符

看样子strong修饰符编译器就能完美进行内存管理，但遗憾的是仅仅通过strong修饰符还不能解决循环引用问题。循环引用容易发生内存泄漏。所谓内存泄漏就是应当废弃的对象在超出其生存周期后继续存在。在持有对象的弱引用时，若该对象被废弃，则此弱引用将自动失效且处于nil.

__unsafe_unretained修饰符

附有__unsafe_unretained修饰符的变量不属于编译器内存管理的对象，如果没有确保其确实存在，那么应用程序就会崩溃

__autoreleasing修饰符

指定@autoreleasepool块来替代 “autoreleasepool类对象的生成、持有以及废弃”

id的指针或对象的指针在没有显式指定时会被附加上autoreleasing修饰符。虽然可以显式指定autoreleasing修饰符，但在显式指定__autoreleasing修饰符必须注意对象的变量要为自动变量(包括局部变量、函数以及方法参数)

要通过将对象赋值给附加了autoreleasing修饰符的变量来替代autorelease方法，ARC有效时要用@autoreleasepool替代NSAutoreleasePool类，对象赋值给附有autoreleasing修饰符的变量等价于在ARC无效时调用对象的autorelease方法，即对象被注册到autoreleasepool中.编译器会检查方法名是否以alloc/new/copy/mutableCopy开始，如果不是则自动将返回值的对象注册到autoreleasepool中.

由于return使得对象变量超出其作用域,所以该强引用对应自己持有的对象会被自动释放,但该对象作为函数的返回值,编译器会自动将其注册到autoreleasepool

为什么在访问附有weak修饰符的变量时必须访问注册到autoreleasepool的对象呢？这是因为weak修饰符支持有对象的弱引用，访问该对象的过程中该对象可能被废弃，如果把访问的对象 注册到autoreleasepool中,那么在@autoreleasepool块结束前都能确保对象存在.
